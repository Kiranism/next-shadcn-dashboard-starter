# B2B-реферальная иерархия — аудит готовности

> **Контекст:** клиент-производитель БАДов, продаёт через сеть спортклубов-партнёров.  
> Структура: **Руководитель → Менеджеры → Тренеры/Блогеры → Клиенты**. Реф-ссылки выдаются только тренерам/блогерам, выше них — менеджеры и руководители, которые получают свой % с продаж и видят статистику подчинённых.  
>
> **Дата аудита:** 2026-05-23  
> **Статус:** ~70% уже реализовано в виде «персональных планов комиссий» (декабрь 2025 — май 2026).

---

## TL;DR

Базовая инфраструктура для трёхуровневой реф-цепочки **уже есть и работает в продакшене**. На неё навешена логика «зафиксированный план для каждого приглашённого» — то что нужно для B2B (менеджер уволится → его клиенты продолжают приносить руководителю те же %).

**Чего не хватает** для готового продукта под этого клиента:
1. Явных ролей `DIRECTOR / MANAGER / TRAINER / CLIENT` — сейчас все плоско в `User`
2. Ограничения «реф-ссылка только у тренера/блогера»
3. Личного кабинета для тренера/менеджера/руководителя — сейчас всё видит только админ проекта
4. Бот-команд `/моя_команда` для менеджеров и руководителей с просмотром статистики подчинённых
5. Автогрантов на просмотр (сейчас гранты раздаются по одному вручную)

Оценка: **5–8 дней** до production-ready продукта под этот кейс.

## 1. Что уже есть и работает

### 1.1 Доменная модель (Prisma)

```prisma
// Активная тарифная сетка для конкретной ветки
model ReferralCommissionPlan {
  id             String
  projectId      String
  name           String                         // "Спортклубы (3 уровня)"
  maxPayoutDepth Int      @default(3)           // глубина выплат, 1..10
  levels         ReferralCommissionPlanLevel[]
  // ...
}

model ReferralCommissionPlanLevel {
  planId   String
  level    Int        // 1, 2, 3
  percent  Decimal    // 7.0, 2.0, 1.0
  isActive Boolean
}

// Зафиксированный план для каждого приглашённого
model ReferralAttribution {
  userId           String   @unique
  projectId        String
  referrerId       String
  commissionPlanId String
  locked           Boolean  @default(true)
}

// Право видеть статистику чужого юзера
model ReferralStatsGrant {
  projectId     String
  subjectUserId String   // чью статистику смотрят
  viewerUserId  String   // кто смотрит
  @@unique([projectId, subjectUserId, viewerUserId])
}

model User {
  // ...
  outboundReferralPlanId String?  // план, применяемый к ТЕМ, кого пригласил этот юзер
  referralAttribution    ReferralAttribution?  // план, по которому этого юзера приглашали
}

model Project {
  // ...
  referralPlansEnabled            Boolean  @default(false)
  defaultReferralCommissionPlanId String?  // фолбэк-план
}
```

## 2. Чего не хватает под B2B

### 2.1 Роли пользователей

**Сейчас:** в `User` только `currentLevel: String` (бонусный уровень: «Базовый/Серебряный/Золотой») и `metadata: Json`. Нет признака «это тренер/менеджер/директор».

**Последствия:**
- Любой клиент потенциально может стать реферером (если узнает свой `userId`)
- В UI и боте нельзя отличить внутреннего партнёра от обычного покупателя
- Нельзя строить отчёты «выплаты только менеджерам»

**Что нужно:**
- Enum `UserRole { CLIENT, TRAINER, MANAGER, DIRECTOR }` (или `PartnerRole`)
- Поле `User.partnerRole` (default `CLIENT`)
- В `findReferrer` отфильтровывать `role !== CLIENT`

### 2.2 Реф-ссылка только у партнёров

Сейчас `ReferralService.findReferrer` ищет любого активного пользователя по `id` — то есть формально любой может вставить чужой `userId` в `utm_ref` и стать реферером.

**Что нужно:** в `findReferrer` добавить условие `partnerRole IN ('TRAINER', 'MANAGER', 'DIRECTOR')`. Или проще — отдельный флаг `User.canReferOthers: Boolean`, чтобы не привязываться к конкретному набору ролей.

### 2.3 Личный кабинет партнёра

**Сейчас:**
- Один UI — `/dashboard/...` для админа проекта (через `AdminAccount`)
- Тренер/менеджер/директор — это `User` без своего входа в админку
- Видит свою статистику только в Telegram-боте через переменные

**Что нужно решить (продумай со мной):**

#### Вариант 1 — «всё в Telegram-боте» (быстрый, рекомендую первым этапом)

Все партнёры взаимодействуют только через бота. Менеджер — это юзер с `partnerRole = MANAGER`, ему добавляются:

- Команда/кнопка `/моя_команда` — список тренеров (своих рефералов)
- Команда/кнопка `/моя_статистика` — суммарно по всей ветке
- Команда `/выплаты` — история начислений ему
- Тапнул на тренера → видит статистику тренера: его клиентов, оборот, начисленные тренеру бонусы

Плюсы: ни одной новой страницы, переиспользуем workflow + переменные. Можно написать reusable workflow «личный кабинет партнёра» и подключить ко всем проектам через `bot-templates`.

Минусы: ограничен возможностями бота. Длинные таблицы неудобны.

#### Вариант 2 — отдельный портал `/partner`

- Создаём новую сущность `PartnerAccount` или переиспользуем `User` + флаг `canLogin`
- Логин по email/phone + OTP в Telegram
- Своя страница `/partner` с урезанным набором: «Моя ссылка», «Мои клиенты», «Моя команда», «Выплаты»
- Менеджер дополнительно видит «Команда тренеров»; директор видит «Все менеджеры + все тренеры + все клиенты»
- Доступы строятся на `ReferralStatsGrant` + автогенерация по `referredBy`-цепочке

Плюсы: масштабируемо. Хорошо продаётся как премиум-фича.

Минусы: реальные 5+ дней работы. Новая аутентификация.

#### Вариант 3 — гибрид

Бот + минимальный публичный личный кабинет на mobile-friendly странице (без логина, доступ по уникальному токену в реф-ссылке). Например, тренер открывает `gupil.ru/p/<token>` — видит свою статистику. Менеджер открывает свой токен — видит расширенный отчёт.

Без логина = меньше работы. Защита через подписанный токен в URL.

> 💡 **Моё предложение:** делать **Вариант 1** в рамках этого MVP. Всё что просит клиент — «менеджер видит своих тренеров» — закрывается через бота. Если бизнес взлетит, через 1–2 месяца добавим Вариант 2 как платный апгрейд.

### 2.4 Автогранты на просмотр

Сейчас `ReferralStatsGrant` создаётся вручную через API. Если у директора 50 менеджеров и 500 тренеров, нужно явно создать 550 грантов.

**Что нужно:** правило по умолчанию «партнёр уровня N автоматически видит свою ветку вниз». Двa варианта:

1. **Денормализация**: создать гранты сразу при назначении роли. Минус: нужно поддерживать актуальность (юзер сменил реферера → пересчитать гранты).
2. **Вычисление на лету в API/боте**: вместо `referralStatsGrant.findFirst` делать обход цепочки `referredBy` и пускать любого, кто стоит выше viewer. Плюс: ничего поддерживать не надо. Минус: больше запросов.

Я бы шёл через вариант 2 (вычисление). Гранты оставить как ручной механизм для исключений.

### 2.5 UI визуализации иерархии

Уже есть наполовину готовая спека `.kiro/specs/user-referrals-display/` (описывает дерево рефералов в профиле юзера). Её нужно:
- Доделать до production
- Добавить переключение «вверх/вниз» (показывать кто пригласил вверх, кого пригласил вниз)
- Добавить колонки «оборот», «начислено», «выплачено комиссии»

### 2.6 Назначение outbound-плана через UI

Сейчас в панели «Планы» есть текстовое поле `User ID` куда нужно вписать cuid руками. Для B2B нужно:
- Селектор пользователей с поиском по фамилии/телефону/email (использовать существующий `cmdk` Command-компонент)
- Фильтр по `partnerRole`
- Bulk-операция «назначить план всем тренерам клуба X»

### 2.7 Множество клубов в одном проекте (если нужно)

Если у клиента 1 производитель → много клубов-партнёров с независимыми руководителями, понадобится сущность `Organization` (клуб) + `User.organizationId`. Тогда руководитель = директор клуба, видит только свой клуб.

**Если клиент один директор на всю организацию** — `Organization` не нужна. Можно отложить.

### 2.8 Прочие мелочи

| Что | Почему важно |
|---|---|
| Лимит depth = 3 для UI | Сейчас слайдер 1..10, для клиента нужен «нативный» 3 |
| Скрыть legacy `ReferralLevel` UI когда `referralPlansEnabled = true` | Чтобы не путать настройщика двумя источниками правды |
| Уведомления партнёру о новом подопечном | Менеджер хочет знать «у тебя новый тренер Иван» |
| Уведомления о начисленной комиссии | «Вам начислено 350 ₽ за покупку клиента Мария на 5000 ₽» |
| Экспорт в Excel ведомости выплат | Для бухгалтерии партнёра-производителя |
| ИНН/название юрлица для тренера/менеджера | Для оформления выплат самозанятым/ИП |

## 3. Сценарий «как заведётся» этот клиент сейчас (без доработок)

Можно прямо сегодня, без новых таблиц. Будет работать, но кривовато:

```
1. Включить ReferralProgram в проекте
2. Включить referralPlansEnabled
3. Создать план «Спортклуб»: L1=7%, L2=2%, L3=1%, depth=3
4. Назначить его дефолтным
5. Создать руководителя как обычного юзера, дать ему utm_ref ссылку
6. Руководитель вручную регистрирует менеджеров через свою ссылку
7. Каждому менеджеру — назначить outbound-план «Спортклуб» (или другой если %% отличаются)
8. Менеджеры регистрируют тренеров через свои ссылки (тоже outbound = Спортклуб)
9. Тренеры дают свои ссылки клиентам
10. Клиент покупает → автомат идёт по цепочке: тренер 7% → менеджер 2% → директор 1%
11. Каждый видит у себя в боте «приглашено N, бонусов M»
```

**Что не получится:**
- Менеджер не сможет в боте посмотреть **статистику конкретного тренера** (только агрегаты по себе)
- Любой клиент случайно может стать реферером (если узнает чей-то userId)
- Нет UI «список моих тренеров» — менеджеру придётся самому помнить кого приглашал
- Если тренера выгнали — нужно вручную обнулить его outbound-план (иначе он продолжит приглашать)

То есть для **прототипа/пилота** хватит. Для **продажи как продукта** нужны доработки из секции 2.
