# Документирование

## Структура документации
- `/docs/api.md` — документация API endpoints
- `/docs/webhook-integration.md` — гайд по интеграции
- `/docs/telegram-bots.md` — настройка ботов
- `/docs/database-schema.md` — схема БД
- `/docs/changelog.md` — хронология изменений
- `/docs/tasktracker.md` — трекер задач

## Правила размещения документов (анти-беспорядок)

> Добавлено 2026-06-14 после уборки 86 эфемерных файлов из корня.

### 🚫 ЗАПРЕЩЕНО
- Создавать `*.md`-отчёты в корне проекта (`MOYSKLAD_*`, `INSALES_*`, `*_FIX.md`, `*_COMPLETE.md`, `*_SUMMARY.md`, `READY_*`, `PUSH_*`, `DEPLOY_*` и т.п.).
- Дублировать статусные/«готово!»-отчёты — они быстро устаревают и засоряют репозиторий.
- Класть тестовые HTML-страницы-стенды в `public/` (они публично деплоятся). Используйте локальные временные файлы вне `public/`.

### ✅ ОБЯЗАТЕЛЬНО
- Вся постоянная документация живёт в `docs/` (или `user-docs/` для пользовательской), единственный `*.md` в корне — `README.md`.
- Прогресс и статусы фиксируются в `docs/changelog.md` и `docs/tasktracker.md`, а не в отдельных файлах-отчётах.
- Гайды по интеграциям — один файл на интеграцию в `docs/` (например `docs/moysklad-direct-api-integration.md`), без `*_FIX`/`*_FINAL`/`*_QUICKSTART` спутников.
- При добавлении нового гайда в `docs/` — добавить ссылку в индекс `docs/README.md`.
